# BUG-2026-05-30T170200: Article images broken; admin logout does not return to landing page

## Problem

Two related UX bugs reported after loading Danilo Albergaria articles:

### 1. Featured images not showing (public site)

- **Actual:** Article cards and featured hero show broken-image placeholders on `/` and `/artigos`. HTML `<img>` tags are rendered with valid-looking Appwrite URLs, but the browser cannot load them.
- **Expected:** Featured images from Appwrite Storage display on the homepage, article list, and article detail pages.
- **Reproduce:**
  1. Ensure articles exist with `featuredImage` file IDs (migration or admin upload).
  2. Open `http://localhost:4321/` or production site.
  3. Observe broken thumbnails and hero image.

### 2. Admin logout does not redirect to landing page

- **Actual:** Clicking **Sair** in the admin sidebar calls `/api/auth/logout` then navigates to `/admin/login`.
- **Expected:** After logout, the user should land on the public homepage (`/`), not remain in the admin auth flow.
- **Reproduce:**
  1. Log in at `/admin/login`.
  2. Click **Sair** in the admin sidebar.
  3. Browser goes to `/admin/login` instead of `/`.

## Root Cause Analysis

### Images — verified

**Phase 1 (Reproduce):** Confirmed on live data. Sample file `6a1b4131001bd200998d` in bucket `images` returns:

- `GET .../files/{id}/preview?...` → **403** with body: _"Image transformations are blocked on your current plan."_
- `GET .../files/{id}/view?project=...` → **200** (valid JPEG)
- `GET .../files/{id}/download?project=...` → **200**

**Phase 2 (Isolate):** Public components (`ArticleCard`, article detail page) build image URLs via `getImageUrl()` in the Appwrite helper module. That helper always uses the **`/preview`** endpoint (server-side resize/transform). The bucket has `read("any")` and `fileSecurity: false`, so permissions are not the blocker — the **preview/transform API tier** is.

**Phase 3 (Hypothesis):** The bug occurs because the app depends on Appwrite image transformations (`/preview`), which are unavailable on the current Appwrite plan, instead of serving raw files via `/view`.

**Alternatives considered (lower probability):**

- Missing `PUBLIC_*` env vars at SSR — ruled out; URLs are constructed with correct endpoint/project/bucket but still 403 on preview.
- Missing file read permissions — ruled out; `/view` succeeds with empty file-level permissions because bucket grants `read("any")`.

**Phase 4 (Verify):** Direct `fetch` against preview vs view endpoints confirmed hypothesis. Risk: **Low** (URL construction change only; CSS already handles sizing).

### Logout redirect — verified

**Phase 1 (Reproduce):** Admin layout client script posts to logout API then sets `window.location.href = "/admin/login"`.

**Phase 2 (Isolate):** Failure is in the admin layout logout handler, not the logout API (API correctly clears session cookie).

**Phase 3 (Hypothesis):** Post-logout navigation target is hardcoded to the login page instead of the public landing page.

**Phase 4 (Verify):** Code inspection of admin layout confirms. No E2E covers logout redirect today. Risk: **Low**.
