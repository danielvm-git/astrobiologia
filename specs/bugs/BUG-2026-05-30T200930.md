# BUG-2026-05-30T200930: Article translation fields not translating or saving

## Problem

When an admin translates an article to all locales via DeepL and saves:

- **Actual:** Some fields remain in Portuguese (silent failure), and translated slugs are not persisted when updating existing translation documents.
- **Expected:** All five text fields (`title`, `excerpt`, `content`, `metaTitle`, `metaDescription`) are translated via DeepL, errors surface to the user, and all fields including `slug` persist on save.

**Reproduce:**

1. Edit an article with Portuguese content filled in
2. Switch to EN (or any non‚Äìpt-br locale) tab
3. Click "Traduzir com DeepL"
4. If DeepL fails, fields silently stay in Portuguese
5. Save the article ‚Äî slug may not persist for existing translation rows

## Root Cause Analysis

### Prior bug context

Related to [BUG-2026-05-30T190400](./BUG-2026-05-30T190400.md) (metaDescription validation on save) ‚Äî that fix normalized string fields but did not address translation or slug persistence. This is a **separate, novel issue**.

### Bug 1 ‚Äî Slug dropped on update (verified)

**Phase 2 ‚Äî Isolate:** The article update API computes `uniqueSlug` for each translation but the `updateDocument` path for existing rows omits `slug` from the payload. Create path includes slug; update path does not.

**Phase 4 ‚Äî Verify:** Unit test confirms `updateDocument` payload lacks `slug` before fix; passes after adding `slug: uniqueSlug`.

**Risk level: Medium**

### Bug 2 ‚Äî Silent translation failure (verified)

**Phase 2 ‚Äî Isolate:** The client translation helper calls `/api/admin/translate` but never checks `res.ok`. On 503/500, the response body is `{ error: "..." }` with no `translated` field. The helper falls back to original Portuguese text via `?? text`, so the UI shows untranslated content with no error.

**Phase 4 ‚Äî Verify:** Unit test mocks non-OK fetch ‚Äî before fix, promise resolves with PT text; after fix, promise rejects and `ArticleEditor` catch block can alert the user.

**Risk level: Medium**

## Resolution

**Fixed:** 2026-05-30

**Root cause confirmed:** Update path omitted `slug`; translate helper swallowed API errors by returning source text.

**Fix applied:**

- Added `slug: uniqueSlug` to translation update payload in article API
- `translateText` throws on non-OK response or missing `translated` field

**Hardening added:** Unit tests for slug persistence, translate error propagation, and full five-field happy path.

**Evidence:** 69/69 unit tests pass; typecheck clean.

**Verify command:** `pnpm --filter @astrobiologia/web-astro test:unit && pnpm --filter @astrobiologia/web-astro check`
